Migrating from V1 to V2

Timeline

Starlink V1 API will be deprecated May 1, 2026

All v1 endpoints should be expected to return errors after that date. As Starlink has grown the V1 API authorization model of linking service accounts to users has shown its shortcomings. The Starlink V2 API establishes a new auth paradigm of linking V2 service accounts to the Starlink account instead, and will be maintained for future API versions. We understand this deprecation of V1 incurs significant migration pain, but we believe that the security benefits of V2 are worth it.

Auth

The V2 API biggest change from V1 is that a single service account can only be used to make API calls for the unique Starlink account it is linked to. This means that any integration that requires calling the Starlink API for multiple different Starlink accounts will need to employ logic to maintain and refresh multiple bearer tokens, and attach them to requests as needed.

Service accounts

Integrations with the Starlink V1 API were able to utilize a single V1 service account to make calls to any Starlink account that the creating user was a contact on. In V2 this requires creating separate V2 service accounts for every Starlink account that you want to interact with the API for. The process of creating V2 service accounts can only occur on the Starlink.com settings page .


Endpoint Migration Guide

The exact specifications of all V2 endpoints can be found in the API reference section. This guide aims to highlight key changes from v1 endpoints to their equivalent v2 endpoint.

Note that all V2 endpoints are relative to the single Starlink account that is linked to the bearer token attached on requests. As a result the /accounts/{accountNumber} prefix is not present on any V2 endpoints. All pagination limit parameters were removed in v2. The default page size is 100 unless otherwise specified.

Base path change: As part of the release of V2 API Starlink has changed the request path from web-api.starlink.com/enterprise/ to starlink.com/api/public. Starlink will continue to support the legacy path until the deprecation of V1 API on May 1, 2026, but all integrations with V2 API should use the new path to avoid issues.

  • Telemetry stream: The v1 path of web-api.starlink.com/telemetry/stream/v1/telemetry will only be maintained until the deprecation date. Instead when migrating to V2 streaming use the new path of starlink.com/api/public/v2/telemetry/stream.

GET v1/accounts

V2 closest version: GET /v2/account (Get account information)

Differences:

  • Request:
    • All request parameters were removed
  • Response:
    • This now returns account info for the single Starlink account the token is linked too, instead of a paginated list of accounts.

POST v1/accounts/{accountNumber}/billing-cycles/query

V2 closest version: POST v2/data-usage/query (Get data usage)

Differences:

  • Request:
    • serviceLinesFilter body parameter changed to serviceLineNumbers
    • queryStartDateParam body parameter removed
    • pageIndex body parameter moved to query parameter named page (v2/data-usage/query?page=1)
    • pageLimit body parameter removed
  • Response: unchanged

GET v1/account/{accountNumber}/service-lines/available-products

V2 closest version: GET v2/products (Get products)

Differences:

  • Request:
    • limit query parameter removed. Page size is 100 in v2

GET v1/account/{accountNumber}/addresses

V2 closest version: GET v2/addresses (Get all addresses)


GET v1/account/{accountNumber}/addresses/{addressReferenceId}

V2 closest version: GET v2/addresses/{addressReferenceId} (Get address)


POST v1/account/{accountNumber}/addresses/{addressReferenceId}

V2 closest version: POST v2/addresses (Create address)


PUT v1/account/{accountNumber}/addresses

V2 closest version: PUT v2/addresses/{addressReferenceId} (Update address)

Differences:

  • Request:
    • addressReferenceId body parameter moved to be query parameter

POST v1/account/{accountNumber}/addresses/check-capacity

No equivalent v2 endpoint. Creating or updating service lines will check capacity internally and return error messages if there isn't sufficient capacity.


GET v1/account/{accountNumber}/routers/{routerId}

V2 closest version: GET v2/routers/{routerId} (Get router)


GET v1/account/{accountNumber}/routers/configs

V2 closest version: GET v2/routers/configs (Get all router configs)


GET v1/account/{accountNumber}/routers/configs/{configId}

V2 closest version: GET v2/routers/configs/{configId} (Get router config)


POST v1/account/{accountNumber}/routers/configs

V2 closest version: POST v2/routers/configs (Create router config)


PUT v1/account/{accountNumber}/routers/configs/{configId}

V2 closest version: PUT v2/routers/configs/{configId} (Update router config)


PUT v1/account/{accountNumber}/routers/batch-config

V2 closest version: POST v2/routers/configs/assign (Router config assignment)

Differences: V2 config assignment is all done using a single endpoint that allows assigning a specific config (or none) to a list of routers. All V1 config assignment operations can be accomplished using the V2 endpoint. See V2 config assignment endpoint for exact request format.


PUT v1/account/{accountNumber}/routers/{routerId}/config

V2 closest version: POST v2/routers/configs/assign (Router config assignment)

Differences: V2 config assignment is all done using a single endpoint that allows assigning a specific config (or none) to a list of routers. All V1 config assignment operations can be accomplished using the V2 endpoint. See V2 config assignment endpoint for exact request format.


DELETE v1/account/{accountNumber}/routers/{routerId}/config

V2 closest version: POST v2/routers/configs/assign (Router config assignment)

Differences: V2 config assignment is all done using a single endpoint that allows assigning a specific config (or none) to a list of routers. All V1 config assignment operations can be accomplished using the V2 endpoint. See V2 config assignment endpoint for exact request format.


PUT v1/account/{accountNumber}/update-default-router-config

V2 closest version: PUT v2/routers/configs/default (Set default router config)


GET v1/account/{accountNumber}/router-local-content

V2 closest version: GET v2/routers/local-content (Get list of router local content files)

Differences:

  • Response:
    • fileMD5Hash field renamed to fileContentHash to align with the field name in router config HTTPS server section

POST v1/account/{accountNumber}/router-local-content

V2 closest version: PUT v2/routers/local-content (Upload router local content file)

Differences:

  • Response:
    • Now returns object containing fileContentId and fileContentHash for the uploaded file, which are the values that can be put in the router config HTTPS server section to specify local content.

GET v1/account/{accountNumber}/routers/sandbox/clients

V2 closest version: GET v2/routers/sandbox/clients (Get sandbox clients)


POST v1/account/{accountNumber}/routers/sandbox/clients

V2 closest version: POST v2/routers/sandbox/clients (Update sandbox clients)


POST v1/account/{accountNumber}/routers/sandbox/{sandboxId}

No equivalent V2 version.


PUT v1/account/{accountNumber}/routers/sandbox/heartbeat

V2 closest version: PUT v2/routers/sandbox/heartbeat(Update sandbox heartbeat)


POST v1/account/{accountNumber}/routers/{routerId}/reboot

V2 closest version: PUT v2/routers/{routerId}/reboot (Reboot router)


PUT v1/account/{accountNumber}/update-default-router-tls-domain

No equivalent V2 version at this time.


GET v1/account/{accountNumber}/routers/tls-config

No equivalent V2 version at this time.


POST v1/account/{accountNumber}/routers/tls-config

No equivalent V2 version at this time.


DELETE v1/account/{accountNumber}/routers/tls-config

No equivalent V2 version at this time.


GET v1/account/{accountNumber}/service-lines

V2 closest version: GET v2/service-lines (Get all service lines)

Differences:

  • Request:
    • limit query parameter removed. Page size is 100

GET v1/account/{accountNumber}/service-lines/{serviceLineNumber}

V2 closest version: GET v2/service-lines/{serviceLineNumber} (Get service line)


POST v1/account/{accountNumber}/service-lines

V2 closest version: POST v2/service-lines


DELETE v1/account/{accountNumber}/service-lines/{serviceLineNumber}

V2 closest version: DELETE v2/service-lines/{serviceLineNumber} (Create service line)


PUT v1/account/{accountNumber}/service-lines/{serviceLineNumber}/nickname

V2 closest version: PUT v2/service-lines/{serviceLineNumber}/nickname (Update service line nickname)


PUT v1/account/{accountNumber}/service-lines/{serviceLineNumber}/product/{productReferenceId}

V2 closest version: PUT v2/service-lines/{serviceLineNumber}/product

Differences: The PUT and POST versions of this V1 endpoint do the same thing but one supports specifying data blocks. For V2 these have been combined to a single endpoint, PUT v2/service-lines/{serviceLineNumber}/product

  • Request:
    • productReferenceId query parameter moved to be body parameter

POST v1/account/{accountNumber}/service-lines/{serviceLineNumber}/product/{productReferenceId}

V2 closest version: PUT v2/service-lines/{serviceLineNumber}/product (Update service line product)

Differences: The PUT and POST versions of this V1 endpoint do the same thing but one supports specifying data blocks. For V2 these have been combined to a single endpoint, PUT v2/service-lines/{serviceLineNumber}/product

  • Request:
    • productReferenceId query parameter moved to be body parameter

PUT v1/account/{accountNumber}/service-lines/{serviceLineNumber}/recurring-data

V2 closest version: PUT v2/service-lines/{serviceLineNumber}/data/recurring (Set recurring data blocks on service line)


POST v1/account/{accountNumber}/service-lines/{serviceLineNumber}/top-up-data

V2 closest version: POST v2/service-lines/{serviceLineNumber}/data/top-up (Add one time top up data block to service line)


POST v1/account/{accountNumber}/service-lines/{serviceLineNumber}/opt-in

V2 closest version: POST v2/service-lines/{serviceLineNumber}/data/opt-in (Opt in priority data)


DELETE v1/account/{accountNumber}/service-lines/{serviceLineNumber}/opt-out

V2 closest version: POST v2/service-lines/{serviceLineNumber}/data/opt-out (Opt out priority data)


GET v1/account/{accountNumber}/service-lines/{serviceLineNumber}/billing-cycle/partial-periods

V2 closest version: GET v2/service-lines/{serviceLineNumber}/billing-cycles/partial-periods (Get billing partial periods)


POST v1/account/{accountNumber}/user-terminals/{userTerminalId}/{serviceLineNumber}

V2 closest version: POST v2/service-lines/{serviceLineNumber}/user-terminals (Add user terminal to service line)

Differences:

  • Request:
    • userTerminalId query parameter replaced with body parameter deviceId which can instead be a userTerminalId, Kit #, or serial number.

DELETE v1/account/{accountNumber}/user-terminals/{userTerminalId}/{serviceLineNumber}

V2 closest version: DELETE v2/service-lines/{serviceLineNumber}/user-terminals (Remove user terminal from service line)

Differences:

  • Request:
    • deviceId query parameter replaced with body parameter deviceId which can still be a userTerminalId, Kit #, or serial number.

POST v1/account/{accountNumber}/service-lines/{serviceLineNumber}/public-ip

V2 closest version: POST v2/service-lines/{serviceLineNumber}/public-ip (Set service line public IP setting)


GET v1/account/{accountNumber}/user-terminals

V2 closest version: GET v2/user-terminals (Get all user terminals)

Differences:

  • Request:
    • limit query parameter removed. Page size is 100
    • active query parameter removed, this caused confusion due to how Starlink returned historical records of UTs that were no longer on the account when active was false. Can still filter by user terminals that are on/not on service lines using the hasServiceLine query parameter.

POST v1/account/{accountNumber}/user-terminals/{deviceId}

V2 closest version: POST v2/user-terminals (Add user terminal to account)

Differences:

  • Request:
    • deviceId query parameter moved to body parameter deviceId

DELETE v1/account/{accountNumber}/user-terminals/{deviceId}

V2 closest version: DELETE v2/user-terminals/{deviceId} (Remove user terminal from account)


POST v1/account/{accountNumber}/user-terminals/{deviceId}/reboot

V2 closest version: POST v2/user-terminals/{deviceId}/reboot (Reboot user terminal)


POST v1/account/{accountNumber}/user-terminals/batch-config

V2 closest version: POST v2/user-terminals/configs/assign (User terminal config assignment)

Differences:

  • Request:
    • deviceId body parameter renamed to userTerminalIds to make clear this field needs to be specifically userTerminalIds, making all uses of deviceId in V2 refer to fields that can be a userTerminalId, kit #, or serial number
  • Notes: user terminal configIds are not queryable via the API today, configIds must be found on the Starlink website.
